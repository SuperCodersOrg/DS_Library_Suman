## Problem Encountered

After completing the architectural design, the next challenge was converting the design into a functional application. The primary difficulty involved implementing a command-processing engine that continuously reads user input, tokenizes commands, validates their syntax, and dispatches them to the appropriate database operation while maintaining a clean and modular architecture.

Another challenge involved handling runtime exceptions generated by the underlying `HashMap`. Operations such as retrieving or deleting a non-existent key throw exceptions, which would terminate the application if left unhandled. Since Redis Lite is designed as an interactive database, the application should continue accepting commands even after an invalid operation.

Presenting stored data also required additional consideration. The existing `printBuckets()` function of `HashMap` was intended for debugging the internal bucket organization rather than displaying database contents in a user-friendly format. Therefore, a dedicated function was required to display all stored key-value pairs in a structured tabular format without exposing the internal bucket arrangement.
